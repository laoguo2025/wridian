# 蒸馏 · Author Craft Skill

> 女娲造人，大神造法。这里蒸馏的不是作者本人，而是他在多部作品中反复使用、可迁移、可检验的创作决策系统。

## 核心定位

`zhengliu-skill` 是 `huashu-nuwa` 的创作类作者专用进化版。

它不以公网访谈、社交媒体、传记材料为主源，而以知识库内已经完成的作品拆解为主源：

```text
02拆解报告/{类型}/{作者名}/《作品名》/
```

目标产物不是“像某作者那样说话”，而是一个可运行的创作顾问 skill：

- 这个作者如何设计故事？
- 他反复解决什么叙事问题？
- 他的核心专利是什么？
- 哪些方法跨作品复现？
- 哪些只能在特定题材/篇幅/读者预期中成立？
- 别人拿到 skill 后，能否不依赖本地文件直接使用？

## 与女娲的关键差异

| 维度 | 女娲 `huashu-nuwa` | 蒸馏 `zhengliu-skill` |
|---|---|---|
| 主源 | 著作、访谈、长文、公开表达 | 多部作品拆解文件 |
| 蒸馏对象 | 思想家/专家/人物/主题 | 小说作者、编剧、网文大神、内容创作者 |
| 证据单位 | 原文观点、公开表态、决策记录 | 作品机制、情节结构、人物弧、情绪节奏、章节功能 |
| 输出重点 | 人物认知操作系统 | 创作决策操作系统 |
| 可分发要求 | sources 可随 skill 打包 | 必须嵌入自包含作品案例，不依赖本地路径 |
| 禁忌 | 编造此人没说过的话 | 把单部作品技巧误判为作者稳定模型 |

## 触发条件

当用户表达以下意图时，启动本 skill：

- “大神蒸馏”
- “蒸馏某作者”
- “从这些作品拆解里提炼作者 skill”
- “做一个猫腻/马伯庸/辰东/南派三叔/甲鱼/卖报/肘子的创作 skill”
- “更新某大神 skill”
- “把 02拆解报告里的某作者作品集中蒸馏”
- “这个作者的创作方法能不能做成 skill”

不适用：

- 只分析单部作品 → 用 `chaijie-skill`
- 只是模仿文风 → 拒绝降级为仿写器，改为提炼“可迁移创作机制”
- 没有作品拆解也没有作品文本 → 信息不足，先要求补拆解或先跑 `chaijie-skill`

## 基本原则

1. **从作品出发，不从作者名声出发**
   只把拆解中反复出现的机制提炼为模型。

2. **跨作品复现才叫心智模型**
   单部作品出现的是技巧；两部以上出现的是倾向；三部以上跨题材复现才可能是创作 OS。

3. **案例必须自包含**
   对外发布版 skill 不写“见 02拆解报告/xxx”。必须截取并压缩成读者能独立理解的作品案例证据。

4. **不做本地路径依赖**
   内部制作可以读取 `02拆解报告`，最终 skill 不依赖任何 `/Users/...` 或知识库本地路径。

5. **不神化大神**
   每个模型必须写失效条件。作者的强项通常伴随盲区。

6. **不替用户写正文**
   输出创作判断、结构建议、检查清单、重构方向；除非用户明确要求，否则不代写大段文本。

## 作者 skill 化闸门

蒸馏产物必须是可运行的小 skill，不是作者研究报告。最终 `SKILL.md` 必须回答：

| 问题 | 最低要求 |
|---|---|
| 输入是什么 | 用户带着什么创作问题、素材、大纲、章节或题材来调用 |
| 处理逻辑是什么 | 作者式判断步骤、心智模型、决策启发式或操作公式 |
| 输出结果是什么 | 诊断、结构方案、人物方案、改写建议、题材迁移路线或检查清单 |
| 谁买单/用于什么场景 | 创作者、编剧、短剧团队、小说项目在什么阶段使用 |
| 失效边界是什么 | 不适用题材、媒介、篇幅、读者预期和误用风险 |

以下内容不得通过：

- 只介绍作者风格、履历、代表作或读后感。
- 把单部作品技巧写成作者铁律。
- 没有自包含案例证据，要求使用者回本地 `02拆解报告` 查证。
- 没有操作步骤，只剩“这个作者很会写人物/铺垫/情绪”等空话。
- 最终 skill 保留 `/Users/...`、临时目录或内部拆解路径依赖。

# 执行流程

## Phase 0：确认蒸馏对象与范围

收到作者名后，确认四件事：

1. 作者是谁；
2. 要蒸馏全部作品，还是某一阶段/题材；
3. 目标用途：创作顾问、结构诊断、题材迁移、已有 skill 更新；
4. `02拆解报告` 中是否已有该作者 ≥2 部作品拆解。

执行纪律：如果用户已经说“可以”、表达授权、或用催促语要求“开工/别问/直接做”，不要再次确认或复述任务；按默认范围直接执行，先产出文件或清单，再简短汇报。除代码、路径、文件名、英文专名外，面向用户的回复用中文，不使用英文进度标签。

用户没有额外限定时，默认：

```text
范围：02拆解报告中该作者全部已拆解作品
用途：创作顾问 + 结构诊断 + 题材迁移
输出：08大神蒸馏/{author}.skill/SKILL.md
```

## Phase 1：定位作品拆解语料

在知识库中定位该作者的作品拆解目录。

优先路径：

```text
02拆解报告/小说/{作者名}/
02拆解报告/网文/{作者名}/
02拆解报告/剧本/{作者名}/
02拆解报告/影视/{作者名}/
02拆解报告/{其他类型}/{作者名}/
```

如果目录命名不一致，用作品名和作者名交叉搜索。

### 批量盘点作者候选

当用户问“哪些作者达到 N 部”“哪些作者可以蒸馏”“优先蒸馏谁”时，直接扫描 `02拆解报告` 下作者目录：以包含 `README.md` 或综合报告的 `《作品名》` 子目录计为已拆作品。列出达到门槛的作者、作品数、作品名，并同时检查 `08大神蒸馏/**/SKILL.md` 判断已有/未有作者 skill。

盘点后应写入或更新 `08大神蒸馏/大神蒸馏优先级清单.md`，并给出“新建优先”和“优化已有”两组队列。若用户已经授权“可以”，不要停在清单上，默认从最高优先级作者开始新建或优化一个 skill。

新建或优化作者 skill 后，同步更新 `08大神蒸馏/大神索引.md`，至少记录作者、skill 目录、状态、更新时间和样本边界；未安装到外部 Agent 时标注“已蒸馏未安装”。

### 批量连续蒸馏闭环

当用户只说“继续”“继续蒸馏”或在上下文压缩后要求恢复推进时，默认按 `大神蒸馏优先级清单.md` 的“新建优先”队列继续，不要重复汇报已经完成的长清单。每完成一位作者，必须执行闭环：

1. 写入 `{author}.skill/SKILL.md` 与 `版本记录.md`。
2. 更新 `大神索引.md`，状态标为“已蒸馏未安装”。
3. 更新 `大神蒸馏优先级清单.md`：该作者改为“已有 xxx.skill / 已完成初版”，并从“新建优先”队列移除或顺延。
4. 运行自检脚本，验证 frontmatter、核心 section、自包含案例、本地路径依赖、索引/优先级同步。
5. 将稳定完成状态写入 `fact_store`，只记作者级成果，不记录流水账。
6. 若上下文预算和用户授权仍允许，可继续处理下一位优先作者；最终汇报只给本轮增量结果。
7. 当“第一批，新建优先”被清空后，不要停在“暂无”；按“第二批，优化已有”队列转入扩容/优化已有作者 skill，优先处理清单中明确标注“已有 skill 偏短”“补证据/失效域/细分模型”的作者。

执行坑点与本地经验见 `references/batch-author-distillation-notes.md`。该参考文件还包含批量连续蒸馏后的索引/优先级顺延检查、精确数量自检脚本和队列残留排查。

### 最低语料标准

| 可用作品数 | 处理方式 |
|---:|---|
| 0 | 不蒸馏；先跑 `chaijie-skill` 拆作品 |
| 1 | 只能生成“单作品技法卡”，不能称为大神 skill |
| 2 | 可生成初版 skill，但所有模型标注“两部验证/样本有限” |
| 3-4 | 可生成稳定 skill，标注跨作品证据 |
| ≥5 | 可提炼作者创作 OS，并做阶段进化分析 |

## Phase 2：按创作维度重读拆解文件

### 大语料读取纪律

进入作品拆解前，先做文件数量、大小、行数、命中数预检；不得把整批源文件、超大单文件或大段扫描结果一次性塞进主上下文。默认用脚本在本地过滤，只回传少量命中片段和精炼结论。若只是补现有 skill 的缺口，先读现有 `SKILL.md`、`版本记录.md`、索引和优先级清单并跑闭环自检；自检能定位缺口时，不要重新批量读取全部源材料。

不要按女娲的“著作/对话/表达/他者/决策/时间线”六路调研。大神蒸馏改为六个创作维度：

| 维度 | 读取目标 | 提取问题 |
|---|---|---|
| 1 故事引擎 | 综合报告、情节结构、卷纲 | 作品靠什么持续推进？任务、谜题、关系、制度、系统、代价？ |
| 2 人物系统 | 人物轨迹、人物原型、关系表 | 主角/反派/配角的功能如何分配？谁驱动谁？ |
| 3 信息释放 | 悬念、伏笔、揭密、反转记录 | 作者如何控制读者知道什么、不知道什么、何时知道？ |
| 4 情绪调度 | 高潮、虐点、爽点、喜剧、沉默段 | 情绪如何蓄力、释放、降格、反刍？ |
| 5 世界/制度/规则 | 世界观、制度、系统、规则、职业体系 | 规则如何成为情节发动机，而不是背景设定？ |
| 6 表达与场景语法 | 句式、对白、场面、动作、节奏 | 作者如何让一个场景“像他”？ |

每个维度输出候选发现：

```markdown
## 候选机制

### 机制名
- 出现作品：
- 具体案例：
- 可迁移规则：
- 可能失效：
- 置信度：高/中/低
```

## Phase 3：跨作品筛选

把候选机制分成四级：

| 等级 | 标准 | 输出位置 |
|---|---|---|
| 心智模型 | ≥3部作品复现，且能解释多个创作选择 | 核心心智模型 |
| 核心专利 | ≥2部作品复现，操作步骤清晰 | 核心专利/操作公式 |
| 决策启发式 | 某类场景下反复出现的判断规则 | 决策启发式 |
| 单作技法 | 只在一部作品突出成立 | 作品案例/参考技法，不上升为作者 OS |

筛选标准：

1. **跨作品复现**：至少两个作品存在机制同构。
2. **生成力**：能指导一个新故事怎么写。
3. **排他性**：不是所有作者都会这么做。
4. **失效可描述**：能说清楚什么时候不好使。
5. **案例可压缩**：能用 100-200 字讲清一个作品证据。

质量分级：

| 等级 | 处理 | 判定 |
|---|---|---|
| S | 写入作者 skill 核心模型 | 2 部以上复现，有操作公式、自包含案例、边界和使用场景 |
| A | 写入参考技法或样本边界 | 有启发但样本少、步骤不稳、只适合局部场景 |
| B | 丢弃 | 单作感想、风格形容、证据不足、不能指导新创作 |

## Phase 4：生成自包含案例证据

最终 skill 不引用本地路径。每条核心模型都要内嵌作品案例。

### 案例写法

错误写法（仅作为禁止示例，不要复制到最终作者 skill）：

```markdown
证据：见「本地作品拆解目录」中的某篇分析笔记
```

正确写法：

```markdown
### 《长安的荔枝》：制度性死局

任务表面是送荔枝，真正压力来自官僚系统、时间限制和责任转嫁。阻力不是某个坏人，而是一整套人人都能自辩、合在一起却压迫主角的制度链。

**可迁移规则**：不要只设计“坏人阻挠”；设计一个所有人都按规则行事、但结果仍然压迫主角的系统。
```

### 案例密度标准

| 内容 | 案例要求 |
|---|---|
| 每个心智模型 | 2-3 个作品案例 |
| 每条核心专利 | 至少 1 个作品案例 |
| 每条决策启发式 | 至少 1 个应用场景或作品片段 |
| 每个失效域 | 至少说明一种误用后果 |

## Phase 5：构建蒸馏 skill

输出到：

```text
08大神蒸馏/{author}.skill/SKILL.md
```

标准结构：

```markdown
---
name: {author}-perspective
description: {作者}创作心智 skill：用于{3-6个核心用途}。
---

# {作者}·创作心智 SKILL v{版本}

> 不是模仿文笔，是加载他的创作决策框架。

## 触发场景
## 核心身份
## 使用原则
## 回答工作流
## 一、心智模型
## 二、决策启发式
## 三、核心专利 / 操作公式
## 四、作品案例证据（自包含版）
## 五、表达DNA
## 六、知识边界与失效域
## 七、与其他作者的区分
## 八、从零开始的操作序列
## 九、质量验证记录
```

## 回答工作流模板

```markdown
## 回答工作流

### Step 1：判断问题类型

| 类型 | 特征 | 行动 |
|---|---|---|
| 结构诊断 | 用户已有故事设定/大纲/章节 | 用心智模型检查主引擎、人物、信息释放、情绪节奏 |
| 从零设计 | 用户只有题材/梗概 | 先跑底层追问，再给结构方案 |
| 题材迁移 | 用户想把该作者方法用于非原题材 | 保留机制，替换载体，并标注失效风险 |
| 对比选择 | 用户不知道该用哪个作者视角 | 与其他作者模型对比，说明适用边界 |
| 作品分析 | 用户问某作品为什么有效 | 回到作品案例证据，拆机制而非夸风格 |

### Step 2：跑作者式底层追问

列出 3-5 个该作者最核心追问。

### Step 3：给出创作判断

输出格式：

1. 一句话诊断；
2. 适用的心智模型；
3. 作品案例证据；
4. 可执行修改；
5. 失效风险。
```

## Phase 6：质量验证

必须做 8 项检查：

| 检查项 | 通过标准 |
|---|---|
| frontmatter | 有 `name` 和 `description` |
| 自包含 | 最终作者 skill 没有绝对本地路径、临时目录、内部拆解文件引用等不可分发依赖 |
| 案例证据 | 有“作品案例证据（自包含版）” section |
| 边界 | 有“知识边界/失效域/诚实边界” |
| 验证分级 | 区分三部验证、两部验证、单部参考 |
| 输入 | 写清用户拿什么来调用 |
| 处理逻辑 | 有心智模型、决策启发式、操作公式或工作流 |
| 输出 | 写清能产出什么创作判断或方案 |

### 禁止项

- 不把单部作品技巧写成“作者铁律”。
- 不用“这个作者很会写人物/很会铺垫”这种空话。
- 不生成不能落地的抽象鸡汤。
- 不在最终 skill 中保留本地路径。
- 不把表达风格模仿放在创作机制之前。
- 不把作者 skill 做成静态知识卡或作品资料卡。

## 深化版蒸馏：先选载体，再选能力

当用户要求“深化版”“短剧向深化”“小说向深化”时，不要把同一个作者核心特点机械拆成短剧版和小说版。正确顺序是：

1. 先读取通用作者 skill，形成该作者的核心能力池。
2. 再判断目标载体：短剧、小说、长篇剧本、影视剧等。
3. 从核心能力池中，为该载体挑选最适合深化的一项能力，而不是把某一项能力硬套所有载体。
4. 短剧向优先选择：高情绪、强冲突、快反馈、主角在场、可表演、可分集钩子、场面化释放的能力。
5. 小说向优先选择：长线伏笔、叙述视角、内心沉积、缺位影响、意义回收、章节追读和阅读耐心能承载的能力。
6. 若用户指出作者本人有具体创作关系或行业用途，例如“师父一直带我们写短剧剧本”，应把该背景纳入选点判断，但最终仍以作品机制和载体适配为准。

例：师父的通用能力池里同时有延迟情绪、缺位叙事、最小物件、选择权、开放终点等。短剧向深化不应优先选“缺位叙事”，因为主角长期缺位会削弱观看；应优先选“延迟情绪”，让情绪在可拍场面中蓄压、迟爆、回收。小说向则可以优先深化“缺位叙事”，因为小说能承载不在场人物、沉默线索、记忆反刍和长线意义回收。

## 更新已有大神 skill

当用户说“更新 XX 大神 skill”：

1. 读取旧 `SKILL.md`；
2. 定位新增作品拆解；
3. 只检查新增作品是否：
   - 强化旧模型；
   - 推翻旧模型；
   - 暴露失效域；
   - 产生新专利；
4. 更新版本号；
5. 新增或替换自包含案例；
6. 不重写全部，除非旧结构不合格。
