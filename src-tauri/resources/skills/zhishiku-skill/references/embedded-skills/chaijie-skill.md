# 拆解 Skill · 02拆解报告工作流

> 拆解不是读后感。拆解是把一部作品变成可检索、可复盘、可蒸馏、可再创作的案例资产。

## 核心定位

`chaijie-skill` 负责把单个作品或单个案例沉淀进：

```text
02拆解报告/
```

它和两个 skill 分工如下：

| Skill | 负责什么 | 输出 |
|---|---|---|
| `chaijie-skill` | 拆单部作品/案例 | `02拆解报告` 案例工作区 |
| `tilian-skill` | 从案例继续提炼知识卡 | `03-07` 知识卡 |
| `zhengliu-skill` | 从同一作者多部作品拆解中蒸馏作者创作心智 | `08大神蒸馏/{author}.skill` |

优先级：

- 用户要拆单部作品 → 用 `chaijie-skill`。
- 用户要从拆解中提炼卡片 → 可接 `tilian-skill`。
- 用户要蒸馏作者 → 先确认 `02拆解报告` 中该作者已有多部作品，再用 `zhengliu-skill`。

## 触发条件

启动本 skill 的典型表达：

- “拆解这部小说/剧本/电影/短剧”
- “录入《作品名》”
- “把这个作品放进 02拆解报告”
- “分析这个案例，后面要蒸馏作者”
- “补拆某大神的一部作品”
- “重拆/精读/案例分析/做综合报告”
- “为大神蒸馏准备素材”

不适用：

- 用户只问某个故事观点 → 直接回答或检索已有资料。
- 用户要把多部已拆作品蒸馏成作者 skill → 用 `zhengliu-skill`。
- 用户要写知识卡而不是拆作品 → 用 `tilian-skill`。

### 复合指令处理

用户经常在一条消息中同时要求拆解+提炼，或同时要求拆解多部作品：

- "拆解并提炼 X" → 先完成 chaijie-skill 全流程，再无缝接入 tilian-skill Phase 5。
- "拆解 A 和 B"（同一作者两部）→ 并行建目录+提取，然后分批启动精读分析（受 max_concurrent=3 限制，可能需要分两批）。
- "提炼 A" + "拆解并提炼 B"（同一作者）→ A 走 tilian 快速路径，B 走完整流程，最后合并做交叉验证评级。

# 执行流程

## Phase 0：确认作品与目标

先判断用户给的是：

| 输入 | 处理 |
|---|---|
| 已有源文件路径 | 直接读取/拆分 |
| 作品名但无源文件 | 先在知识库搜索；找不到则提示需要源文本/剧本/链接 |
| 片段文本 | 建立片段案例或询问是否属于某作品 |
| 多部作品 | 按作者建立多个作品目录，逐部拆；不要混成一份报告 |
| 只说作者名 | 不是拆解任务，转 `zhengliu-skill` 或要求指定作品 |

如果用户说“给后续大神蒸馏用”，拆解时额外加强：

- 反复技法候选；
- 作者级创作习惯；
- 与该作者其他作品可对比的机制；
- `大神蒸馏候选.md`。

## Phase 1：源类型判断

```text
源是？
├─ 长篇小说（>10万字）→ 分卷并行精读
├─ 中短篇小说（2-10万字）→ 通读或2路并行
├─ 短篇/片段（<2万字）→ 直接写短拆报告
├─ 剧本/影视 → 按幕/场/集拆分，做视听或场景功能拆解
├─ 短剧 → 按集和钩子拆，重点拆爽点密度与留存节点
├─ 游戏剧情 → 按章节/任务/副本线拆
├─ 方法论书籍 → 按章节拆论点链
├─ 文章/营销文案 → 拆说服链和传播结构
├─ 音视频 → 先转录，再按文本处理
└─ 图片/短视频 → 秒级/画面级拆解
```

## Phase 1.5：拆解质量闸门

拆解产物是后续知识卡的上游模具。宁可少产候选，也不能把弱证据包装成强结论。

### 证据锚点

所有关键判断必须绑定证据锚点：

```text
长篇小说：章节/分卷/源文件 + 100字内短摘或场景定位
剧本影视：集数/场号/时间点 + 动作或对白短摘
短剧：集数/卡点/留存节点 + 画面或对白短摘
方法论/文章：章节/小标题/段落位置 + 原句短摘
```

无证据锚点的内容只能写入“观察/疑问”，不得进入知识卡候选、大神蒸馏候选或作者规律。

### 覆盖率声明

每份分析笔记和综合报告开头必须声明：

```text
阅读方式：全文/分卷全读/抽样/降级采样
覆盖范围：已读章节或场次
缺失范围：缺章、乱码、广告、低质源、未读段落
结论权限：全局判断/阶段判断/局部观察
```

抽样或降级采样只能产出“局部观察”和“待复核候选”；不能直接写成全书机制、作者稳定技法、S级知识卡候选。

### 候选分级

所有“知识卡候选”先分三级：

| 等级 | 含义 | 后续动作 |
|---|---|---|
| A 可提炼候选 | 多处证据支撑，可迁移，有边界 | 交给 `tilian-skill` 继续评级 |
| B 仅报告候选 | 有启发但证据窄、像案例总结或题材惯例 | 留在报告，不入库 |
| C 禁止提炼 | 无锚点、抽样不足、源文本缺失、剧情复述、作者规律证据不足 | 不得交给提炼 |

### 源文本缺陷限制

若源文本存在缺章、乱码、广告污染、关键段落缺失或只读 sample：

- 必须在 README/报告/笔记中标注缺陷范围。
- 缺陷区间不得生成技法、人物弧、情绪曲线或作者规律。
- 关键高潮/转折/结局缺失时，只能交付资料整理和补源建议，不能产出综合机制结论。

### 分段与降级

- 长篇先按卷/册/情节阶段分段；最多 3 路并行。
- 子代理读取超时或爆上下文时，可以采样或生成 sample，但必须写入覆盖声明。
- 任何采样、mini_sample、只读摘要生成的报告，都只能产出局部观察或待复核候选。
- epub/mobi 结构变体、Python 路径、批量提取经验查 `references/batch-epub-extraction.md`。

批量拆多书时，先批量归一化源文件，再按 3 路分批精读；同作者多书的候选交给 `tilian-skill` 合并交叉验证。

## Phase 2：建立案例目录

创建或复用目标目录。不要把不同作品混在同一文件夹。

小说默认：

```text
$ROOT/02拆解报告/小说/{作者名}/《作品名》/
```

如果作者未知：

```text
$ROOT/02拆解报告/小说/作者待确认/《作品名》/
```

目录内建议保留：

```text
source/                 # 可选，源文件较多时使用
分卷/                   # 长篇拆分文件
分析笔记-*.md
《作品名》-综合报告.md
《作品名》-审美反应.md
《作品名》-大神蒸馏候选.md
```

## Phase 3：拆分与精读

先把 txt/epub/mobi/pdf/docx 归一化为可定位的 `_source/chapters`、`_source/分卷`、`manifest.md` 和原始源文件。具体提取脚本与格式陷阱查 `references/batch-epub-extraction.md`。

归一化验收：

- 章节数、总字数、首中尾抽查合理。
- 源文件质量问题已记录到 README/元数据。
- 分卷边界不切断关键高潮。
- 后续分析只基于已声明的覆盖范围。

### 长篇小说

按自然结构切分：卷/册 > 情节阶段 > 事件块。

原则：

- 每份 1.2-3 万行为宜。
- 小卷合并，超大卷拆分。
- 不在高潮中间切。
- 推荐 2-4 路并行。

每份分析笔记必须包含：

```markdown
# 分析笔记-{范围}

## 0. 覆盖声明
- 阅读方式：
- 覆盖范围：
- 缺失范围：
- 结论权限：

## 一、事件流
| 序号 | 事件 | 功能 | 情绪 | 后果 | 证据锚点 |

## 二、人物轨迹
| 人物 | 本段目标 | 行动 | 关系变化 | 弧光变化 | 证据锚点 |

## 三、信息释放
| 信息/伏笔 | 首次出现 | 本段变化 | 是否回收 | 读者效果 | 证据锚点 |

## 四、技法拆解
- 技法名：
- 证据锚点：
- 生效原因：
- 可迁移规则：
- 反例/边界：
- 候选等级：A可提炼/B仅报告/C禁止提炼

## 五、审美反应
- [高光] 场景 → 好在哪
- [低光] 场景 → 差在哪

## 六、大神蒸馏候选
- 可能是作者稳定技法：
- 只属于本作的单作技巧：
- 后续需与哪些作品对比：
```

### 剧本/影视/短剧

重点拆：

- 场景目标；
- 冲突推进；
- 视听调度；
- 钩子位置；
- 集尾/幕尾留存；
- 人物关系变化。

短剧额外记录：

```markdown
## 留存节点
| 时间/集数 | 钩子 | 情绪 | 是否承接下一集 |
```

### 方法论/文章

重点拆：

- 论点链；
- 证据链；
- 说服顺序；
- 概念命名；
- 可迁移方法。

## Phase 4：综合报告

每个作品必须有综合报告。

标准结构：

```markdown
# 《作品名》综合报告

## 1. 一句话判断
这部作品真正有效/失效的核心机制是什么。

## 2. 基础信息
- 类型：
- 作者/主创：
- 篇幅/集数：
- 拆解范围：
- 阅读方式：
- 缺失范围：
- 结论权限：
- 拆解日期：

## 3. 故事引擎
作品靠什么持续推进。

## 4. 人物系统
主角、反派、配角、关系网如何分工。

## 5. 情节结构与节奏
关键阶段、高潮、反转、留白。

## 6. 信息释放与悬念
伏笔、揭密、误导、读者认知控制。

## 7. 情绪曲线
爽、虐、燃、怕、笑、沉默、反高潮。

## 8. 世界观/制度/规则
规则如何推动情节。

## 9. 可复用技法清单
| 技法 | 证据锚点 | 生效原因 | 可迁移方式 | 边界 | 候选等级 |

## 10. 知识卡候选
按 `03-07` 分类列出。必须标 A/B/C；只有 A 可交给 `tilian-skill` 继续评级。

## 11. 大神蒸馏候选
| 候选机制 | 证据锚点 | 结论权限 | 后续对比作品 |

## 12. 失效点与低光
哪些地方不成立，为什么。

## 13. 后续动作
- 是否提炼知识卡：是/否
- 是否进入大神蒸馏样本：是/否
- 需要补拆：
```

## Phase 5：大神蒸馏候选文件

如果作品属于某位待蒸馏作者，创建：

```text
《作品名》-大神蒸馏候选.md
```

结构：

```markdown
# 《作品名》大神蒸馏候选

## 作者级候选机制
| 机制 | 证据锚点 | 可迁移规则 | 结论权限 | 是否需跨作品验证 |

## 单作技巧
只在本作特别成立，暂不上升为作者模型。

## 与作者其他作品的对比问题
- 这个机制是否在其他作品出现？
- 载体是否变化但功能相同？
- 是否存在反例？

## 可供 zhengliu-skill 使用的自包含案例
### 案例标题
100-200字说明机制。

**证据锚点**：章节/场号/集数/位置。
**可迁移规则**：一句话。
**结论权限**：局部观察/阶段判断/全书机制。
```

这个文件的作用：让 `zhengliu-skill` 后续不用回读全量笔记，也能快速抽取作者级案例。

## Phase 6：知识卡提炼接口

拆解完成后，不强制生成 `03-07` 卡。根据用户目标决定：

| 用户目标 | 动作 |
|---|---|
| 只做案例工作区 | 停在 `02` |
| 要沉淀知识卡 | 调用 `tilian-skill` 或按其 card-formats 写卡 |
| 要蒸馏作者 | 累积到 ≥2 部作品后调用 `zhengliu-skill` |

传给 `tilian-skill` 的候选必须满足：

- 候选等级为 A。
- 至少 2 个独立证据锚点；若只有 1 个锚点，必须标“仅报告候选”。
- 不是剧情复述、人物性格词、题材常识或单一爽点描述。
- 抽样/降级采样得出的候选必须先标“待复核”，不得直接建议入库。

如果生成知识卡，必须遵守 `SCHEMA.md`：

- `03故事模型`
- `04人物原型`
- `05情节方程`
- `06写作技法`
- `07综合素材`

`02` 的分析文件不替代知识卡。

## Phase 7：自检

完成前检查：

- 案例目录是否在 `02拆解报告`。
- 源文件是否已放入案例目录或能定位。
- 是否有至少一份分析笔记。
- 分析笔记是否有覆盖声明和证据锚点。
- 是否有综合报告。
- 综合报告是否区分 A/B/C 候选。
- 若为大神样本，是否有 `大神蒸馏候选.md`。
- 抽样/缺章/低质源是否限制了结论权限。
- 是否没有把知识卡误放进 `02`。
- 如果生成了 `03-07` 卡，是否通过 `health-check.sh`。

交付格式：

```markdown
完成：
- 案例目录：
- 源文件：
- 分析笔记：N份
- 综合报告：
- 大神蒸馏候选：有/无
- 知识卡：未生成/新建N/更新N
- 候选分级：A/B/C 数量
- 覆盖声明：全文/抽样/降级采样
- 自检：通过/待处理
```

## 与 zhengliu-skill 的衔接

当同一作者已有 ≥2 部作品完成拆解，提醒：

```text
该作者已有 N 部作品拆解，可进入 zhengliu-skill 做作者创作心智蒸馏。
```

但不要自动蒸馏，除非用户明确要求。

## Pitfalls

主流程只保留质量闸门和常规路径。低频格式事故、epub/mobi结构变体、Hermes/Python环境坑统一查阅：

- `references/batch-epub-extraction.md`

遇到源文件异常时，先处理源质量和覆盖声明，再写分析结论。

## 参考文件

- `references/batch-epub-extraction.md` — 批量 epub 提取经验（分卷策略、cn_to_int映射、超时处理、epub结构变体）

## 质量底线

- 不写泛泛读后感。
- 不只总结剧情。
- 每个技法都要有具体场景。
- 每个判断都要说明生效原因。
- 不把单部作品技巧上升为作者铁律。
- 不为了产出知识卡破坏 `02` 工作区完整性。
