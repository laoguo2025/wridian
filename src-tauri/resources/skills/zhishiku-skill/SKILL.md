# 知识库总控流程

> AI短剧/竖屏短剧剧本创作或修订也属于知识库调用场景：先按 `references/ai-short-drama-script-usage.md` 检索短剧方法论与案例，再输出剧本格式（集号、场号、人物、△动作/镜头、对白、卡点/AI提示），不要写成小说散文稿。

`zhishiku-skill` 是Wridian知识库操作系统：负责初始化、路由、体检、安装、进化、归档。它必须自带拆解、提炼、蒸馏三大能力的内置镜像；若环境里存在独立 `chaijie-skill`、`tilian-skill`、`zhengliu-skill`，优先加载独立 skill；若独立 skill 不存在，必须读取本 skill 的 `references/embedded-skills/`...

## 核心原则

- **总控包含子能力**：`zhishiku-skill` 内置 `chaijie/tilian/zhengliu` 三套能力镜像；独立子 skill 可作为可维护版本被优先调用，但不能成为硬依赖。
- **默认可分发**：初始化脚本不能依赖本机绝对路径；默认目标是 `~/Desktop/Wridian知识库`，用户指定路径优先。
- **先建结构，再做内容**：没有知识库时先执行初始化，再进入拆解/提炼/蒸馏。
- **08大神蒸馏是 skill 工厂仓库**：`zhengliu-skill` 产出的作者小 skill 保存在这里，并可安装到 Claude Code 或其他 Agent 的 skill 根目录。
- **清理只归档，不物理删除**：默认移动到 `09文件归档/待删除-YYYY-MM-DD/`。物理删除必须二次确认。
- **体检包含进化**：体检不仅找问题，也生成可执行修复；低风险格式修复可自动应用，高风险语义改写需确认。
- **体检报告归档**：每次全库体检必须输出或更新 `00知识库治理/知识库体检-YYYY-MM-DD.md`；报告是 skill 产物，Wridian 只展示，不替代体检判断。
- **冲突先标记**：发现知识卡观点冲突、适用条件冲突或依据不足时，先标记 `体检状态/冲突对象/不确定性`，不自动合并、不自动删除。
- **总控先守质量**：`zhishiku-skill` 不只分发任务，还要阻断低质量拆解、弱证据知识卡和不可分发作者 skill 进入下游。
- **意图不明先澄清**：用户没有明确要拆解、提炼、蒸馏还是体检时，先问任务模式，不默认跑完整链路。

## 总控质量闸门

### 1. 知识卡 skill 化闸门

`03-07` 每张知识卡都按一个小 skill 验收。不能回答下面五项的，不得作为高价值卡入库：

| 问题 | 最低要求 |
|---|---|
| 输入是什么 | 说明什么任务/场景下调用 |
| 处理逻辑是什么 | 给出判断步骤、公式或操作顺序 |
| 输出结果是什么 | 说明能产出诊断、桥段、结构、人物方案还是素材 |
| 谁买单/用于什么场景 | 写清适用创作场景、使用对象或产品用途 |
| 失效边界是什么 | 说明不适用题材、媒介、阶段或误用风险 |

只解释作品、只总结剧情、只给审美感受的内容，只能留在 `02拆解报告`，不能进入 `03-07`。

### 2. 拆解入口闸门

调用 `chaijie-skill` 后，若产物要继续提炼或蒸馏，必须先确认：

- 分析笔记和综合报告有覆盖声明：阅读方式、覆盖范围、缺失范围、结论权限。
- 关键判断有证据锚点。
- 知识卡候选已分 A/B/C；只有 A 可传给 `tilian-skill`。
- 抽样、缺章、低质源只能产出局部观察或待复核候选。

未满足时，任务停在 `02拆解报告`，不得继续写入 `03-07` 或进入大神蒸馏。

### 3. 知识卡入口闸门

调用 `tilian-skill` 后，入库前必须抽查本轮新增/更新卡：

- 能从 `编译来源` 回到具体证据，而不是只回到泛泛报告。
- 有调用摘要、适用边界、关联索引、输入、处理逻辑、输出结果和使用场景。
- 不是剧情复述、人物性格词、题材常识或单一爽点描述。
- A/B 候选不得写入 `03-07`。

### 4. 运营闭环

每次拆解、提炼、蒸馏、安装、体检、清理完成后，必须更新或提醒更新：

```text
00知识库治理/调用记录台账.md
```

记录至少包括：任务类型、产物路径、调用表现、是否返工、下一步进化。体检时要按“质量 × 频率”输出杠杆、浪费、内耗、垃圾四象限，并给出废弃、合并、降评级、补入口、补关联、重写、升金卡等动作。

### 5. 版本迭代机制

体检必须回答“多久没进化了”。按 `updated`、`版本记录.md`、调用记录判断：

| 进化年龄 | 状态 | 动作 |
|---|---|---|
| 0-30 天 | 新鲜 | 正常使用 |
| 31-90 天 | 观察 | 结合调用表现复查 |
| 91-180 天 | 待复查 | 补证据、补场景或合并 |
| >180 天 | 老化 | 高频优先重写，低频合并/封存 |

`03-07` 更新 `updated`；`08大神蒸馏` 作者 skill 必须有同目录 `版本记录.md`；BB 大 skill 由自检脚本和变更说明维护。

## 任务模式澄清

进入入口路由前，先判断用户是否已经说清本次目标。明确意图可直接执行：

- 说“拆解 / 拆这本 / 分析这部作品” → 只跑 `chaijie-skill`，产物进 `02拆解报告`。
- 说“提炼 / 出卡 / 萃取知识卡” → 跑 `tilian-skill`，从已有或本轮合格拆解写入 `03-07`。
- 说“蒸馏 / 大神 skill / 作者 skill” → 跑 `zhengliu-skill`，产物进 `08大神蒸馏`。
- 说“体检 / 检查 / 进化知识库” → 跑 health-check 与对应修复流程。

意图不明确时，只问一句：

```text
这次目标是只拆解、拆解后提炼知识卡、提炼已有拆解，还是蒸馏作者 skill？
```

| 模式 | 动作 | 停止点 |
|---|---|---|
| 只拆解 | 调用 `chaijie-skill` | 停在 `02拆解报告` |
| 拆解 + 提炼 | 先拆解，通过质量闸门后调用 `tilian-skill` | 写入 `03-07` |
| 提炼已有拆解 | 不重拆，从已有 `02` 产物提炼 | 写入 `03-07` |
| 蒸馏作者 skill | 从同作者多部合格拆解蒸馏 | 写入 `08大神蒸馏` |

“全流程”默认解释为“拆解 + 提炼”。只有用户明确要求蒸馏，或同作者已有 2 部以上合格拆解且用户确认，才继续进入作者 skill 蒸馏。

## 源文件归一化与切分

用户提供 `docx/pdf/epub/mobi` 或超长 `md/txt` 时，先读取 `references/source-normalization.md`，完成归一化后再拆解。

流程：

1. 保留原文件到 `01原始资料` 或案例目录 `_source/`。
2. 将复杂格式转为 `md/txt` 中间文件：
   - `docx/epub/html`：优先 pandoc。
   - `mobi`：优先 calibre `ebook-convert` 转 epub，再 pandoc。
   - `pdf`：优先 `pdftotext` 或 OCR，不把 pandoc 当 PDF 主入口。
3. 写 `转换记录.md`。
4. 对中间文件抽查：开头/中段/结尾、章节、中文乱码、PDF 空文本。
5. 文件过大时运行 `scripts/split_text.py` 切成多个 `part-XXX.md/txt`，并生成 `manifest.md`。
6. 后续拆解只读分卷文件，不一次性读取整本超长文件。

推荐切分命令：

```bash
python3 {skill_dir}/scripts/split_text.py \
  --input {转换后的md或txt} \
  --output-dir {案例目录}/_source/分卷 \
  --max-chars 30000
```

质量门槛：转换或切分异常时停止拆解，先修复源文件；不得基于乱码、空文本、缺章文本继续分析。


触发后优先使用独立 `chaijie-skill`；若未安装，读取 `references/embedded-skills/chaijie-skill.md` 执行同等流程。目标产物进入：

```text
02拆解报告/{媒介类型}/{作品名}/
```

### 提炼知识卡

触发后优先使用独立 `tilian-skill`；若未安装，读取 `references/embedded-skills/tilian-skill.md` 以及其格式引用执行同等流程。知识卡只写入 `03-07`，必须遵守：

- S 级才入库。
- A 级留报告候选。
- B 级丢弃。
- 编译来源可定位。
- 关联索引非空。
- 上游 `chaijie-skill` 候选等级必须为 A，且有证据锚点与结论权限。
- 必须像 skill 一样写清输入、处理逻辑、输出结果、使用场景和失效边界。

#### 拆解+提炼收尾验收

完成单部作品的 03-07 入库后，不要只口头汇报，必须执行以下收尾闭环：

1. 针对本轮新增/更新卡逐张自检：frontmatter 可解析、`评审状态: S级保留`、`评级: S`、`编译来源`文件存在、`关联索引`非空且路径存在、板块字段与一级目录一致。
2. 抽查本轮卡片内容：至少 3 张或全部新卡，回源检查证据锚点是否支撑核心断言。
3. 更新该案例 `02拆解报告/.../README.md`：将知识卡提炼状态改为“已完成（N张S级新卡/更新N张）”，追加“已提炼S级知识卡”清单和“校验状态”。
4. 跑全库 `health-check.sh`。若只剩历史旧卡的编译来源定位问题，且本轮新增卡针对性自检 OK，可在 README 和汇报中明确“非本案例新增”，不要把旧账误判成本轮失败。
5. 重新统计 03-07 总卡片数，并同步调用记录台账/长期记忆中的作者进度与总数。
6. 最后再标记任务完成并汇报。

macOS/Hermes 路径注意：如果工具拒绝以含中文的知识库路径作为 `workdir`，用命令内 `cd /Users/mac/Desktop && bash 'Wridian知识库/health-check.sh'` 执行，不要把 `workdir` 直接设为 `/Users/mac/Desktop/Wridian知识库`。

## 作者小 skill 安装规则

安装目标按运行环境分流：

```text
Claude Code 默认：~/.claude/skills/{skill-name}/
其他 Agent：{用户指定的 skill 根目录}/{skill-name}/
```

执行：

```bash
python3 {skill_dir}/scripts/install_skill.py \
  --source {08大神蒸馏中的skill目录} \
  --target-root {目标skill根目录} \
  --knowledge-root {知识库根目录}
```

如果用户没有说明目标环境：

- 当前环境是 Claude Code：默认 `--target-root ~/.claude/skills`。
- 当前环境不是 Claude Code，或用户明确使用其他 Agent：先让用户提供该工具的 skill 根目录，再安装。
- 不要把作者小 skill 写死为依赖 `~/.claude/skills/`；安装位置只作为外部参数。

规则：

- 目标不存在：可直接安装。
- 目标已存在：先备份到 `09文件归档/skill备份-YYYY-MM-DD/`，再覆盖；覆盖前需用户确认。
- 安装后更新：`08大神蒸馏/_安装记录.md`。

## 体检与进化

用户说“体检知识库”或“进化 skill”时，执行三层体检。

### 02拆解报告专项补齐

当任务是“把 02拆解报告补规范 / 补齐缺报告缺笔记 / 清零案例缺口”时，先做 02 专项扫描，不要只依赖全库 `health-check.sh`。全库体检会同时暴露 03-07 旧卡 frontmatter、来源断裂、评审状态等大量老账，容易淹没 02 案例缺口。

专项流程：

1. 遍历 `02拆解报告/**/README.md` 识别案例目录，跳过 `_规范化日志`。
2. 判定报告：文件名包含 `综合报告`、`案例分析报告`、`分析报告`、`总拆解报告`，并排除 `待补拆清单.md`。
3. 判定笔记：文件名包含 `分析笔记`、`事件流`、`结构拆解`、`人物与势力线`、`爽点节奏`、`写作模型`。
4. 同步 README 状态，确保不存在 `综合/案例报告：缺` 或 `分析笔记/事件流：缺` 的陈旧标记。
5. 若用已有报告回填分析笔记，必须在笔记中写明“依据既有报告整理，不伪造源剧本文本；后续精拆需回源文件逐场补强”。
6. 每次写入长 Markdown 后立刻抽读文件开头，检查是否出现字面量 `...[truncated]` 或 `[truncated]`；若出现，说明写入内容被工具层截断，必须重写完整文件。

专项验收可用本地脚本扫描案例数、缺报告、缺笔记、README 陈旧标记和字面量 `[truncated]`；规则保留在本节，长脚本不要塞回主文件。

专项清零后再跑全库体检；若全库体检仍报 03-07 老问题，要在汇报中明确“这不是本次 02 缺口”，不要把它误报为 02 失败。

### A. 知识库体检

检查：

- 00-09 目录是否完整。
- `知识库使用说明.md` 和 `00知识库治理/调用记录台账.md` 是否存在。
- 03-07 frontmatter 是否合规。
- 编译来源是否断裂。
- 关联索引是否为空。
- 评级/评审状态是否异常。
- 06写作技法中的审美法则类卡片是否满足 3 独立作品标准。
- 03-07 知识卡是否具备输入、处理逻辑、输出结果、使用场景和失效边界。
- 最近 02 拆解产物是否具备覆盖声明、证据锚点、候选分级。
- 最近 03-07 新卡是否能回源到具体证据。
- 调用记录台账是否记录近期拆解/提炼/蒸馏动作。
- 调用记录台账能否产出质量 × 频率四象限和进化动作。
- 03-07 知识卡是否存在 `updated`，是否超过 180 天未进化。
- 是否存在 `体检状态: 待核查/有冲突/需合并`、`冲突对象`、`不确定性` 或 `[!contradiction]` / `[!gap]` 标记，并汇总进体检报告。
- 03-07 旧卡 frontmatter 修复经验见 `references/frontmatter-healthcheck-notes.md`。
- **编译来源路径修正**：02拆解报告重组后旧路径失效，批量替换前缀即可修复，详见 `references/frontmatter-healthcheck-notes.md` §7。
- **07 调用摘要/颗粒度补写**：需阅读卡片正文后撰写，详见 `references/frontmatter-healthcheck-notes.md` §8。

可自动修：

- 缺失目录。

- description 触发词轻量补充。
- 本地绝对路径改为相对说明。

- frontmatter 格式字段修复。
- 体检脚本的低风险解析兼容修复：例如支持旧卡合法状态 `S级保留` / `A级待复审`、支持 `编译来源` 的数组/引号/行号范围格式；这类修复优先改校验器，不要为了通过体检丢失卡片业务语义。

#### 03-07 旧卡 frontmatter 体检经验

处理 03-07 旧卡大量“缺字段”时，先判断是否真缺字段，不要盲目补字段。常见根因是 frontmatter 结束符没有独立成行，例如 `---## 一句话定义`、`created: 2026-04-15---`、`updated: 2026-05-07---`。修复方式是只把 `---` 拆成独立行，保留原字段和正文，不改写卡片判断。

若体检报 `评审状态=[S级保留]` 或 `评审状态=[A级待复审]`，先对照 `tilian-skill`：这些是旧卡业务语义中的合法状态。更稳妥做法通常是扩展 `health-check.sh` 的允许值，而不是把所有卡片强行改成 `已确认` / `待评审`。

若体检报 06写作技法 `编译来源` 未定位，检查是否为数组格式或带行号，如 `["02拆解报告/.../报告.md:64-99"]`。校验器应抽取引号内首个路径并去掉末尾 `:行号` / `:起止行` 后再判断文件是否存在，不要删除多来源信息。

需确认：

- 重写作者心智模型。
- 删除旧模型或旧卡。
- 覆盖已安装 skill。
- 合并/拆分 skill。
- 批量移动大量文件。

## 交付格式

完成任务后只报告必要结果，优先中文、短句、结果先行；除代码、路径、文件名、英文专名外，不使用英文进度标签。用户已经说“可以”或催促“开工/别问/直接做”时，下一步必须执行工具取得进展，不再重复确认、复述任务或解释计划。

- 初始化：知识库路径 + 新建/补齐目录数。
- 拆解：案例目录 + 笔记/报告数量。
- 提炼：S 新建/更新数量 + A/B 候选数量。
- 蒸馏：作者 skill 路径 + 是否安装。
- 体检：问题总数 + 最高优先级问题 + 已自动修复项。
- 运营：调用记录是否更新 + 质量闸门是否通过/拦截。
- 清理：归档路径 + 文件数量。
