# 2026-06-13 inline diff and chat editing fixes

## 背景

用户反馈模型执行文件增删改后，右侧回复不应展示文件树操作细节；已有文件内容变更应统一走正文内联 diff；选区胶囊发送后应随消息发送并从输入框清除；批量确认按钮距离过远；模型修改后的回复应点评改前内容；确认修改后需要一次性撤销。

## 变更

- 前端不再把 `fileOperations` 执行明细追加到助手消息正文，保留模型面向用户的正常回复。
- 发送普通输入时在请求发起前清空文本和上下文胶囊，发送快照仍携带胶囊用于消息上下文。
- 空文档新增内容可作为空 target 的 inline diff 插入确认；非空文档仍要求精确原文 target。
- 确认模型 diff 前记录一次性撤销快照；撤销、手动编辑或切换文件后快照失效。
- “全部确认 / 全部取消”改为同一按钮组，避免被容器拉开。
- 后端把模型对当前已打开文件的 `writeFile` 转成 `edits`；对其他已存在文件的 `writeFile` 拒绝执行；新建文件仍允许。
- 共创 prompt 明确：返回 edits 时 reply 点评改前原文或选区并说明修改方案；已有文件内容增删改必须走 edits，新建含内容文件除外。

## 验证

- `npm run build` 通过。
- `cargo fmt` 通过。
- `cargo test file_operation --lib` 在普通 PowerShell 中因缺少 MSVC `cl.exe` 失败；本机常见 `vcvars64.bat` 路径也不存在，未能完成 Rust 测试执行。已新增后端回归测试覆盖当前文件 writeFile 转 inline edit 和已有文件 writeFile 拒绝。

## 回退依据

本轮改动集中在聊天消息组装、正文 diff 守卫、主编辑区状态、共创文件操作协议和 workspace 写入门禁。若需回退，可还原本记录对应提交；后端审计仍保留文件树操作执行记录。
